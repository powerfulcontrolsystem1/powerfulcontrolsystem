Ayuda de APIs

Guia para consumir, documentar y revisar las APIs de Powerful Control System sin romper seguridad multiempresa, permisos, licencias, auditoria ni reglas de negocio. Actualizada el 2026-06-01 con energia solar, roles por modulo y checkout de licencias.

Volver al centro de ayuda Seleccionar empresa

Familias de endpoints

  • /api/public/*: lectura publica controlada, catalogos, portal, visitas, venta publica o documentos publicos con token.
  • /api/empresa/*: datos operativos de una empresa. Requiere sesion, empresa_id, permisos efectivos y licencia cuando aplique.
  • /super/api/*: configuracion global, licencias, empresas, auditoria, comunicaciones y diagnostico del super administrador.

Regla multiempresa

El empresa_id puede venir por query, header o payload, pero el backend siempre debe validar que el usuario autenticado puede acceder a esa empresa. Conocer o editar un ID en URL no concede permisos.

Permisos y licencias

Las rutas empresariales usan wrappers como ventas, inventario, compras, clientes, finanzas, facturacion o seguridad. La licencia puede bloquear un modulo aunque el rol base tenga permiso.

Errores estandar

  • 400: faltan datos o hay payload invalido.
  • 401: falta sesion o credencial valida.
  • 403: sin alcance, permiso o licencia.
  • 404: recurso inexistente o fuera de la empresa.
  • 500: error interno; revisar request_id y logs.

Carritos, estaciones y venta directa

El carrito unificado es la API central para venta directa y estaciones. Venta directa usa el carrito canonico VENTA-DIRECTA-{empresa_id}-0.

GET /api/empresa/carritos_compra?empresa_id=7&include_inactive=1&modo=venta_directa&perm_page=linkVentaDirecta GET /api/empresa/carritos_compra/items?empresa_id=7&carrito_id=123&include_inactive=1 POST /api/empresa/carritos_compra/items POST /api/empresa/carritos_compra?action=pagar_estacion

Para estaciones, enviar estacion_id cuando aplique. Para venta directa, usar modo=venta_directa y perm_page=linkVentaDirecta. El backend debe conservar caja, usuario, abonos, inventario y auditoria por empresa_id.

Energia solar

Modulo empresarial independiente para sistemas solares, baterias, BMS, inversores, alertas y eventos. Usa permisos energia_solar, pagina linkEnergiaSolar y wrapper empresarial.

GET /api/empresa/energia_solar?empresa_id=7&action=dashboard GET /api/empresa/energia_solar?empresa_id=7&action=catalogo GET /api/empresa/energia_solar?empresa_id=7&action=sistemas GET /api/empresa/energia_solar?empresa_id=7&action=alertas&sistema_id=1 GET /api/empresa/energia_solar?empresa_id=7&action=lecturas&sistema_id=1&limit=120 GET /api/empresa/energia_solar?empresa_id=7&action=eventos&sistema_id=1&limit=80 POST /api/empresa/energia_solar?empresa_id=7&action=sistema POST /api/empresa/energia_solar?empresa_id=7&action=alerta POST /api/empresa/energia_solar?empresa_id=7&action=lectura POST /api/empresa/energia_solar?empresa_id=7&action=probar_alerta&sistema_id=1

El rol tecnico_solar solo consulta. Los proveedores base son Victron VRM, SMA Sunny Portal, SolarEdge Monitoring y gateway local; las llaves reales se referencian como env:VARIABLE.

Camaras y DVR

Modulo empresarial independiente para camaras conectadas a DVR/NVR o gateways. Usa permiso camaras, pagina linkCamaras y configuracion por empresa_id.

GET /api/empresa/camaras?empresa_id=7&action=dashboard GET /api/empresa/camaras?empresa_id=7&action=catalogo GET /api/empresa/camaras?empresa_id=7&action=camaras GET /api/empresa/camaras?empresa_id=7&action=camara&id=1 POST /api/empresa/camaras?empresa_id=7&action=camara DELETE /api/empresa/camaras?empresa_id=7&action=camara&id=1

RTSP y ONVIF son fuentes comunes de DVR, pero para verlas en navegador deben pasar por HLS, WebRTC, MJPEG o iframe seguro. Las estaciones pueden usar tipo_estacion=camara y camara_id.

Checklist antes de exponer o cambiar una API

  1. Confirmar wrapper de modulo y accion efectiva por metodo o action.
  2. Validar empresa_id en backend y filtrar todas las consultas por empresa cuando la tabla sea empresarial.
  3. No aceptar IDs cruzados: cliente, producto, carrito, bodega, caja, factura o usuario deben pertenecer a la misma empresa.
  4. Aplicar idempotencia en altas, pagos, licencias, offline, backups o acciones que puedan repetirse por doble clic o red.
  5. No registrar secretos, tokens, certificados, contrasenas ni payloads sensibles en logs, auditoria o documentacion.
  6. Actualizar documentos/api/ayuda_apis.md, OpenAPI generado o contrato tecnico cuando cambie el comportamiento.
